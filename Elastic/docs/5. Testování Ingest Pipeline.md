# 5. Testování Ingest Pipeline

Tento dokument popisuje, jak otestovat a ověřit správné fungování Ingest Pipeline před jejím nasazením do produkčního provozu. Testování zajistí, že se pole (např. `trace.id` nebo `process.id`) extrahují korektně.

---

## 1. Testování v rozhraní Kibana (UI)

Nejjednodušší způsob, jak otestovat pipeline, je přímo v editoru Ingest Pipelines:

1. Přejděte do **Stack Management** -> **Ingest Pipelines**.
2. Klikněte na název vaší pipeline (např. `api-gw-pipeline`).
3. V horním menu zvolte záložku **Test pipeline**.
4. Klikněte na **Add documents**.
5. Do pole **Documents** vložte ukázkový JSON (včetně problémových řádků):
   ```json
   [
     {
       "_source": {
         "message": "2026-03-23 00:00:05,468 INFO      92153 [Global@39]  Activity 'Health' Started - traceId=6f8c64d2-2bbc-4299-a8b2-a2be6bbbf669"
       }
     },
     {
       "_source": {
         "message": "2026-03-23 00:00:05,468 INFO      87887 [Global@39]  Prijat REST API požadavek Health traceparent: traceId:6f8c64d2-2bbc-4299-a8b2-a2be6bbbf669 callerSystem:BDP (client:213.58.181.158)"
       }
     },
     {
       "_source": {
         "message": "2026-03-23 00:00:05,484 INFO      87888 [Global@39]  REST API požadavek Health traceparent: traceId:6f8c64d2-2bbc-4299-a8b2-a2be6bbbf669 úspešne dokoncen"
       }
     }
   ]
   ```
6. Klikněte na **Run the test**.
7. V pravé části uvidíte výsledek – pokud je vše v pořádku, uvidíte pole `processId` s hodnotou `ca2c6f62f2a946548c4b8e0bd75d4a95`.

## 2. Testování přes Dev Tools (API)

Pro pokročilé testování nebo automatizaci můžete použít `_simulate` API. Tento příkaz pošlete v **Dev Tools** (Console).

### Příklad volání:

POST _ingest/pipeline/api-gw-pipeline/_simulate
```json
{
  "docs": [
    {
      "_source": {
        "message": "2026-03-23 00:00:00,031 INFO      92153 [Global@39]  Activity 'Health' Started - traceId=e035baa8-4d74-4bf8-b5bc-0b1ca1c66cc6"
      }
    },
    {
      "_source": {
        "message": "2026-03-23 00:00:15,019 INFO      92816 [Global@18]  Onboarding.VerifyPhoneNumberOtp: Onboarding step set to PhoneVerification in process ca2c6f62f2a946548c4b8e0bd75d4a95"
      }
    }
  ]
}
```

## 3. Co kontrolovat ve výsledku

Při kontrole výsledků se zaměřte na:

- **`correlationId` / `processId`**: Jsou pole přítomna a obsahují správnou hodnotu?
- **`timestamp`**: Odpovídá čas v poli `timestamp` času z logu a je v ISO8601 formátu?
- **`operation`**: Je správně identifikována akce (např. `ResendOtp`)?
- **`requestId`**: Obsahuje ID z logu?
- **`tags`**: Pokud v poli `tags` uvidíte `_grokparsefailure`, znamená to, že Grok pattern neodpovídá formátu logu.

---

## 4. Ladění chyb (_grokparsefailure)

Pokud testování selže:

1. **Zkontrolujte mezery:** Grok je citlivý na počet mezer. Používejte `\s+` pro variabilní počet mezer.
2. **Postupná stavba:** Začněte s jednoduchým patternem (jen `%{TIMESTAMP_ISO8601:timestamp}`) a postupně přidávejte další pole.
3. **Grok Debugger:** V Kibaně (sekce **Dev Tools**) najdete záložku **Grok Debugger**, která je ideální pro rychlé ladění patternů bez nutnosti měnit pipeline.

---

> **Doporučení:** Vždy otestujte alespoň 5-10 řádků různých typů logů (DEBUG, INFO, ERROR), abyste měli jistotu, že pipeline pokrývá všechny varianty.
